## [对美国国家航空航天局阿特米斯计划的批评：高成本、复杂性和低效率](https://idlewords.com/2024/5/the_lunacy_of_artemis.htm)

- 文章批评美国国家航空航天局（NASA）的阿耳忒弥斯计划复杂、成本高，而且与阿波罗任务相比效率低下，尽管技术有所进步。
- 主要的批评意见包括：太空发射系统（SLS）成本高昂、技术落后，猎户座飞船的设计面临挑战，以及使用近直角光环轨道（NRHO）增加了复杂性和风险。
- 在 2026 年之前让人类重返月球的雄心勃勃的时间表被认为是不现实的，可能会出现延误和成本上升，载人登月有可能被永久推迟。

### [反应](https://news.ycombinator.com/item?id=40410404)

- 讨论将阿波罗任务的月球旅行创新解决方案与现代空间探索挑战进行了比较，突出了复杂性和风险。
- 它批判了 NASA 的低效率、官僚作风和过时技术，并与 SpaceX 的创新方法形成对比。
- 对话强调了美国国家航空航天局阿耳特弥斯计划的政治和财政限制，倡导更具成本效益的机器人任务，并探讨了太空探索的动机和益处。

## [3M 公司数十年来一直隐瞒化学家关于人体血液中有毒全氟辛烷磺酸的研究结果](https://www.propublica.org/article/3m-forever-chemicals-pfas-pfos-inside-story)

- 3M 公司的化学家克里斯-汉森（Kris Hansen）于 1997 年发现人体血液中含有全氟辛烷磺酸这种有害化学物质，但她的研究却被向她保证安全的上司制止了。
- 几十年后，美国环保署强制要求清除饮用水中的全氟辛烷磺酸（PFAS）化学物质，揭露了 3M 公司早在 20 世纪 70 年代就知道全氟辛烷磺酸的毒性，但仍在继续生产。
- 尽管为解决 PFAS 污染问题达成了 125 亿美元的和解协议，但全部成本和责任仍未解决，汉森现在志愿从事环境保护工作。

### [反应](https://news.ycombinator.com/item?id=40414316)

- 3M 公司的高管说服科学家约翰逊隐瞒有关全氟辛烷磺酸化学品的有害研究结果，这凸显了一种不鼓励解决此类问题的文化。
- 讨论批评了 3M 公司的保密和道德问题，主张进行独立研究、加强监管和严惩公司的不当行为。
- 该案文强调，需要加强对举报人的保护，并进行系统改革，以追究公司和高管对不道德行为的责任。

## [Uber 成功将数万亿条目分类账数据迁移到自定义分类账存储中](https://www.uber.com/blog/migrating-from-dynamodb-to-ledgerstore/)

- Uber 将超过一万亿条分类账数据从 DynamoDB 迁移到其定制的 LedgerStore (LSG)，以提高成本效益、简化架构并改善性能。
- 迁移涉及 1.2 PB 的不可变记录和 0.5 PB 的二级索引，在不中断业务的情况下确保了数据的完整性和正确性，准确率达到 99.99%。
- 在此过程中，利用 Apache Spark 处理大量数据，并通过速率限制器和保守的推出策略来应对代码错误和 RPC 超时等挑战，在没有停机的情况下完成了为期两年的迁移。

### [反应](https://news.ycombinator.com/item?id=40413891)

- 由于 SQLite 的可扩展性和高写入量限制，将 Uber 1.7 PB 的分类账数据从 DynamoDB 迁移到 SQLite 是不切实际的。
- 讨论批评了 Uber 的成本节约策略，质疑了定制解决方案与 DynamoDB 等商业云服务相比的可行性和隐藏成本。
- 对话强调了高效数据存储的重要性、企业系统迁移的复杂性，以及为吸引客户而提供详尽文档和可靠基准的必要性。

- 讨论强调了加强监管和消费者保护的必要性，建议对企业的不当行为进行更严格的处罚，加强消费者保护法，以及通过小额索赔法院或立法行动进行追索的可能性。

## [将 Raspberry Pi 5 提升至 3.3 GHz：尽管破解了电压，增益仍然微乎其微](https://jonatron.github.io/randomstuff/pivolt/)

- 作者曾尝试在 Raspberry Pi 5 上超频至 3.14 GHz，但发现系统在 3.3 GHz 时并不稳定。
- 为此，作者通过修改系统内存和创建一个服务来管理 CPU 频率和电压，从而规避了 Raspberry Pi 固件规定的 1V 电压限制。
- 尽管达到了更高的时钟速度，但性能提升却微乎其微，这表明所做的努力可能并不值得。

### [反应](https://news.ycombinator.com/item?id=40409718)

- 一位名叫 jonatron 的用户将 Raspberry Pi 5 超频至 3.14 GHz，超过了 Jeff Geerling 之前创下的 3.3 GHz 的记录。
- 讨论强调了 BCM2712 芯片的最佳默认速度为 2.4 GHz，因为在更高速度下，收益会递减，功耗会增加。
- 谈话内容包括冷却方法、与旧 CPU 的性能比较以及 Raspberry Pi 5 的 Linux 主线内核不支持硬件加速等方面的争论。

## [PDF.js 中的严重漏洞允许任意执行 JavaScript](https://codeanlabs.com/blog/research/cve-2024-4367-arbitrary-js-execution-in-pdf-js/)

- Codean Labs 在 Mozilla 的 PDF.js 中发现了 CVE-2024-4367 漏洞，该漏洞允许在打开恶意 PDF 时执行任意 JavaScript，从而带来重大安全风险。
- 该漏洞影响所有火狐用户，可导致 XSS（跨站脚本）攻击和 Electron 应用程序中潜在的本机代码执行。
- 该问题可通过将 PDF.js 更新到 4.2.67 或更高版本、检查依赖关系和调整安全设置来缓解；该问题于 2024 年 4 月 26 日向 Mozilla 披露，修复程序于 2024 年 5 月中旬发布。

### [反应](https://news.ycombinator.com/item?id=40414718)

- CVE-2024-4367 是 PDF.js 中的一个安全漏洞，允许任意执行 JavaScript，可能导致跨站脚本 (XSS) 攻击。
- 讨论强调了网络安全的复杂性，特别是客户端 JavaScript 和不可靠输入的复杂性，并建议在不同来源上隔离 PDF 阅读器。
- 与会者讨论了使用 PDF.js 的安全影响，重点关注 XSS 风险，并建议在嵌入 PDF.js 的应用程序（如 Electron 应用程序）中实施稳健的内容安全策略。

## [提高可读性和可维护性的基本 C 语言编程实践](https://github.com/mcinglis/c-style)

- 作者强调，在 C 语言编程中，正确性、可读性、简洁性和可维护性要优先于速度，主张在优化之前先编写清晰和可维护的代码。
- 主要做法包括使用现代 C 标准（最好是 C11），避免使用非标准扩展名，使用空格而不是制表符，行数控制在 79 个字符以内，以及在代码和文档中保持一致的美式英语。
- 该指南建议不要使用复杂的条件句、开关语句和宏来掩盖控制流，建议使用 `const` 来实现不变性，避免使用无符号类型，并倾向于使用不变的数据结构和纯函数。

### [反应](https://news.ycombinator.com/item?id=40409956)

- C Style：我最喜欢的 C 语言编程实践》的作者讨论了他们对编码风格不断演变的看法，强调机械执行和实时反馈以提高代码质量。
- 关键主题包括平衡可读性和性能、避免过早优化，以及在嵌入式编程中根据上下文做出决定。
- 讨论内容包括编码标准，如 80 字符行限制、制表符与空格，以及使用 Clang sanitizers、管理依赖关系和最小化变量范围等最佳实践。

### [反应](https://news.ycombinator.com/item?id=40414565)

- 讨论围绕 64k 介绍展开，这是一种数字艺术形式，文件大小受限于 64 千字节，强调创作过程而非技术解释。
- 与会者注意到，由于制作 64k 开场白需要耗费大量精力，Demoscene 社区开始转向制作较小的 开场白（4k/1k），并建议录制或直播创作过程，以记录创作历程。
- 对话强调了不断变化的期望和改进的工具，这使得制作高质量 64k 开场白的要求越来越高。
