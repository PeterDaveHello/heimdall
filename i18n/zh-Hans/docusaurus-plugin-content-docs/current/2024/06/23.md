## [“将SSH作为Sudo的替代方案”](https://whynothugo.nl/journal/2024/06/13/ssh-as-a-sudo-replacement/)

- “传统工具如sudo和doas使用setuid二进制文件和权限提升，这在受限的用户命名空间中有局限性，并且需要整个用户会话保留能力。”
- “另一种方法是使用 s6-sudod 将程序分为一个有特权的服务器和一个无特权的客户端，旨在以 root 身份运行命令而不进行特权升级。”
- “作者尝试使用本地SSH进行root命令执行，配置专用的SSH密钥，将sshd绑定到Unix域套接字，并使用ProxyUseFdpass处理套接字文件描述符，从而增强安全性并支持各种身份验证方法。”

### [反应](https://news.ycombinator.com/item?id=40763117)

- “使用SSH作为sudo的替代方案会引入复杂性，因为它需要通过UNIX套接字和非对称加密来进行根二进制文件的通信。”
- “将sudo访问权限限制在wheel组，并确保只有授权用户可以读取或执行它，可以在减少复杂性的同时提供类似的安全性。”
- “像 etckeeper、Ansible 和 Chezmoi 这样的工具可以有效地管理和跟踪配置更改，提供了传统 sudo 使用的替代方案。”

## [“切勿\*使用数据报”](https://quic.video/blog/never-use-datagrams/)

- “在构建互联网应用程序时，TCP因其可靠的传输而被优先选择，而UDP则因其及时性，特别是在实时视频协议中被使用。”
- “直接使用UDP是有风险的，因为需要实现重传和拥塞控制等功能；相反，开发者应该使用QUIC库。”
- “QUIC 和基于 QUIC 的媒体传输(MoQ)支持数据报，但建议开发人员专注于使用 QUIC 流，以避免与数据报相关的陷阱。”

### [反应](https://news.ycombinator.com/item?id=40766079)

- “讨论的中心是使用UDP(用户数据报协议)与TCP(传输控制协议)进行数据传输，重点介绍了每种协议的优缺点。”
- “UDP通常被称为“不可靠”或“尽力而为”，这意味着它不保证传输、顺序或错误检查，不像TCP，它确保可靠、有序和错误检查的传输。”
- “对话中包括了UDP使用案例的例子，例如高频传感器数据和媒体艺术系统，并讨论了使用UDP而非TCP的术语和实际影响。”

## [“所有命令都以逗号开头”](https://rhodesmill.org/brandon/2009/commands-with-comma/)

- “Unix 用户经常在他们的主目录中创建一个 ~/bin/ 目录来存储自定义脚本，但这可能会导致与系统命令的名称冲突。”
- “为了避免这些冲突，作者建议在自定义命令名称前加上逗号，逗号在文件名中被视为普通字符，从而避免冲突。”
- “这种技术结合了标签补全功能，使得浏览自定义命令变得轻松，并且在十多年的时间里被证明是一种可靠的解决方案。”

### [反应](https://news.ycombinator.com/item?id=40769362)

- “一个用户分享了一种在 Windows 上运行 Python 脚本的方法，通过将 python.exe 设置为 .py 文件的默认程序，并将 .py 添加到 %pathext%。”
- “他们在 Linux 上寻求类似的功能，但由于缺乏“关联程序”概念而面临挑战，建议包括使用 shebang(`#!/usr/bin/env python3`)、重命名脚本、创建别名或使用 `update-alternatives` 等工具。”
- “讨论还涉及使用git管理本地配置、使用别名、处理命名空间冲突，以及使用`chezmoi`或ansible等工具组织脚本和配置。”
