## [通过技术提高银行运营效率：对当前挑战和潜在改进措施的分析](https://www.bitsaboutmoney.com/archive/seeing-like-a-bank/)

### [反应](https://news.ycombinator.com/item?id=38180477)

- Patrick McKenzie 仔细检查银行因子系统内沟通和组织不力而导致的运营效率低下，从而引发账户关闭和信用卡欺诈等问题。
- 他指出，银行的分级客户服务结构和某些令人困惑的法律义务（如不披露可疑活动报告）是问题的一部分。
- 麦肯齐认为，改进后的技术可以提高银行的运营能力和客户体验，但他也承认，在整个金融系统的实施可能需要循序渐进。他还强调，有必要重新考虑对客户造成不必要伤害和混乱的具体法规。

## [优化容器中的 Go 应用程序：Linux 调度器和 CPU 限制的作用](https://www.riverphillips.dev/blog/go-cfs/)

- 博文强调，在运行应用程序时，有必要在容器编排器中设置 CPU 限制，以避免使用主机的所有 CPU 能力。
- 深入介绍了 Go 垃圾收集器进程、Linux 调度器的作用，以及通过 GOMAXPROCS 环境变量控制 CPU 线程的重要性。
- Uber 引入了 "automaxprocs "库，它能自动从容器的 cgroups 中计算 GOMAXPROCS 值，从而简化了在容器化应用程序中运行 Go 时的设置。

### [反应](https://news.ycombinator.com/item?id=38181346)

- Go 开发人员在博客中强调了在容器编排器中设置 CPU 限制的重要性，以避免主机的 CPU 消耗过高，因为 Go 运行时往往会使用所有可用的 CPU 能力，从而可能导致高延迟。
- 作者建议使用 GOMAXPROCS 环境变量控制 Go 运行时可创建的 CPU 线程数，该变量与 CPU 配额一致。
- Uber 创建了 "automaxprocs "库，它可以自动从容器的 cgroups 中计算出该环境变量值，从而满足了在容器化应用程序中运行 Go 时合理分配 GOMAXPROCS 的需要。

## [介绍 SectorC：x86 机器引导扇区中世界上最小的 C 编译器](https://github.com/xorvoid/sectorc)

- SectorC 是用 x86-16 汇编语言编写的 C 语言编译器，由于它可以放在 x86 机器的引导扇区中，因此有可能成为最小的 C 语言编译器。
- SectorC 支持大量的 C 语言子集，尽管体积小巧，却能编制复杂的程序，例如涉及移动正弦波动画的程序。
- 虽然 SectorC 被认为是一个小众的使用场景，但对于那些热衷于了解 x86-16 bios 功能和机器模型而不需要大量 x86 汇编知识的人来说，SectorC 不失为一个有用的工具。

### [反应](https://news.ycombinator.com/item?id=38176062)

- SectorC 是一个 C 语言编译器，据说是有史以来开发的最小的编译器之一，采用 x86-16 汇编语言编写。该编译器的独特之处在于，它可以装在 x86 机器的引导扇区中。
- 尽管体积小巧，但 SectorC 支持 C 语言的显著子集，包括函数、if 语句、while 语句和其他功能。这样就可以创建复杂的程序，如移动的正弦波动画。
- 它的开发重点似乎是那些希望了解 x86-16 bios 功能和机器模型的人，为他们提供了一座探索的桥梁，而无需学习大量的 x86 汇编程序。
