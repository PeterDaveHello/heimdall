## [我沒有建構子，但我必須初始化](https://consteval.ca/2024/07/03/initialization/)

- 「在 C++ 中，如果沒有提供建構函式，編譯器可能會生成預設的建構函式，這可能會導致物件未初始化。」
- “對於非類別和非陣列型別，預設初始化不會執行任何操作，而值初始化會將非類別型別初始化為零，然後進行預設初始化。”
- 依賴編譯器生成的建構子可能會導致未定義的行為；建議撰寫自訂建構子以避免這些問題。

### [評論](https://news.ycombinator.com/item?id=40880932)

- 「由於 C++ 是一種多範式的語言，支援嵌入式系統、電子遊戲、伺服器和圖形使用者介面等各種應用，因此它相當複雜。」
- 像 C 和 Rust 這樣的替代語言無法涵蓋 C++ 的所有使用情境，使得 C++ 成為一種獨特但具有挑戰性的語言。
- 雖然 C++ 的複雜性對於性能和靈活性是必要的，但對於經驗較少的團隊來說可能會感到難以應付。

## [「為什麼 Vivaldi 不會跟隨當前的 AI 趨勢？」](https://vivaldi.com/blog/technology/vivaldi-wont-allow-a-machine-to-lie-to-you/)

- Vivaldi 決定不將大型語言模型(LLMs)如 ChatGPT 整合到他們的網頁瀏覽器中，理由是擔心抄襲、隱私侵犯以及生成不準確的信息。
- “LLM 分析大量數據，這可能導致無意中重複受版權保護的材料和洩露私人信息。”
- Vivaldi 旨在提供功能豐富且符合道德標準的瀏覽體驗，並保持對未來以隱私和易用性為優先的機器學習進展的開放態度。

### [評論](https://news.ycombinator.com/item?id=40880322)

- 「Vivaldi 避免整合 AI，特別是大型語言模型(LLMs)的決定，在用戶間引發了重大討論。」
- 「維瓦第立場的支持者強調對大型語言模型(LLM)可靠性和高能耗的擔憂，而批評者則認為LLM提供了對使用者有益的寶貴功能。」
- 這篇解釋 Vivaldi 決定的文章因看起來有偏見且忽視 LLMs 潛在的好處而受到批評，許多人主張應將 AI 整合作為瀏覽器中的可選功能。

## [在 Unison 中編程](https://lwn.net/Articles/978955/)

- Unison 是一種 MIT 授權的程式語言，它將程式碼儲存在資料庫中而非文字檔案中，從而簡化了分散式程式設計。
- 它使用基於雜湊的獨特命名系統，能夠實現無縫的重新命名和版本控制，並具有一個能力系統來追蹤程式碼的依賴性和互動。
- 為雲端運算設計的 Unison 通過 Unison Cloud 提供管理服務，但面臨如缺乏穩定的外部函數介面(FFI)和需要自訂工具等挑戰。

### [評論](https://news.ycombinator.com/item?id=40882133)

- Unison 程式語言以其符合人體工學的設計和代碼在資料庫中的方法而著稱，這種方法通過消除長時間的編譯和對 Git 的需求來簡化編碼。
- 儘管 Unison 是一門年輕的語言，但它提供了強大的工具，包括語言伺服器協定(LSP)整合和一流的文件，這些都提升了開發者的體驗。
- Unison 處理儲存在資料庫中的不可變代碼的方法解決了傳統 REPL 語言中的常見問題，儘管有些使用者擔心現代語言內建了過多工具。
