## [Timeshift：Linux 系統還原工具](https://github.com/linuxmint/timeshift)

- Timeshift for Linux 是一款類似於 Windows 系統還原和 Mac OS Time Machine 的系統備份工具，專注於系統檔案和設定。
- 它支援兩種模式：RSYNC(使用 rsync 和硬連結)和 BTRFS(使用 BTRFS 檔案系統功能)，後者需要特定的子卷佈局。
- 由 Tony George 開發，現已成為 Linux Mint 的 Xapp 專案一部分，Timeshift 提供多種快照層級、跨發行版還原和還原後掛鉤等功能，並提供適用於各種 Linux 發行版的安裝說明。

### [評論](https://news.ycombinator.com/item?id=41039967)

- 「Timeshift 是一款 Linux 的系統還原工具，類似於 macOS 的 Time Machine 和 Windows 的系統還原，允許使用者使用 rsync 和硬連結來創建檔案系統快照。」
- 用戶討論了各種備份解決方案和配置，包括 restic、rclone、ZFS、BTRFS 和 LVM 快照，強調了原子快照對於資料庫一致性和可靠備份的重要性。
- 「這段對話強調了需要強大的備份策略，並比較了像 restic、Borg 和 kopia 這些工具，討論了不同檔案系統和快照方法的優缺點。」

## [2024 年 7 月 Intel Core 第 13/14 代桌上型處理器不穩定報告更新](https://community.intel.com/t5/Processors/July-2024-Update-on-Instability-Reports-on-Intel-Core-13th-and/m-p/1617113#M74792)

- Intel 已經發現其第 13 代和第 14 代桌上型處理器因微碼演算法導致操作電壓升高而產生不穩定性。
- 預計在八月中旬會有一個微碼修補程式來解決此問題，英特爾建議受影響的客戶聯繫支援以獲得協助。
- 用戶表達了對潛在長期 CPU 損壞的擔憂、需要 BIOS 更新的需求，並在尋求管理不穩定性指導的過程中報告了 RMA(退貨授權)問題，直到補丁發布為止。

### [評論](https://news.ycombinator.com/item?id=41039708)

- “英特爾 2024 年 7 月的更新指出，第 13/14 代桌上型 CPU 存在微碼問題，導致電壓請求錯誤，儘管有些用戶懷疑是硬體故障。”
- “由於英特爾延遲解決問題以及有報導指出無故障的 CPU 在未更新微碼的情況下出貨，這可能導致 CPU 長期退化，引發了擔憂。”
- 「Intel 計劃發布微碼補丁，但其效能和對性能的影響尚不確定，用戶對 CPU 穩定性的反應不一。」

## [ASCII 表的優雅](https://danq.me/2024/07/21/ascii/)

- ASCII 表於 1963 年標準化，為 128 個可能的 7 位元二進位碼點中的 100 個賦予了意義，並且至今仍然具有相關性，特別是因為 UTF-8 編碼具有向後相容性。
- 該表格包含控制碼、可列印字元，並遵循特定的二進位模式，其中空格字元是排序用途的第一個可列印字元。
- ASCII 的設計既合乎邏輯又美觀，讓人類學習和理解起來都很順暢，這反映了它的歷史意義和精緻的邏輯。

### [評論](https://news.ycombinator.com/item?id=41040543)

- 「這篇文章討論了 ASCII 表的優雅性和實用性，這是一種用於計算機的字符編碼標準。」
- 它強調了使用者如何透過命令 `man ascii` 在 Linux 系統上存取 ASCII 表，這對於 shell 逃脫碼和正規表示式(regex)非常有用。
- 這段對話包含了歷史背景和個人軼事，講述了通過 manpages 學習各種技術的經歷，並強調了 ASCII 表在計算機領域中長期以來的重要性。

## [「有可能在不進入 FAANG 工作的情況下獲得相同薪水嗎？」](https://news.ycombinator.com/item?id=41042123)

### [評論](https://news.ycombinator.com/item?id=41042123)

- 這些職位通常伴隨著高壓、激烈的競爭或重大的風險和努力，使得它們相比於 FAANG 工作來說較為罕見且不那麼簡單明瞭。

## [終止 OCSP 服務的意圖](https://letsencrypt.org/2024/07/23/replacing-ocsp-with-crls.html)

- 「由於隱私風險和資源效率的考量，Let's Encrypt 將停止支援線上憑證狀態協定(OCSP)，轉而使用憑證撤銷列表(CRLs)。」
- 此變更不會影響網站或訪客，但可能會影響某些非瀏覽器軟體；建議使用者確保其軟體在沒有 OCSP URL 的情況下仍能正常運作。
- CA/瀏覽器論壇的決定使得OCSP對於公開信任的憑證授權機構(CAs)來說是可選的，唯有微軟是例外；一旦微軟也將其設為可選，關閉OCSP服務的時間表將會公佈。

### [評論](https://news.ycombinator.com/item?id=41046956)

- 「Let's Encrypt 計劃停止其 OCSP(線上憑證狀態協定)服務，原因是憑證撤銷列表(CRLs)存在問題以及 OCSP 系統的複雜性。」
- 提案包括使用二進位格式的 CRL 以提高效率，以及強制 OCSP 釘選來解決隱私問題，儘管這可能會干擾許多網站。
- “轉移離開 OCSP 可能會影響非瀏覽器應用程式和嵌入式設備，但使用 Nginx 或 Caddy 等標準網頁伺服器的用戶應該不需要立即進行更改。”
