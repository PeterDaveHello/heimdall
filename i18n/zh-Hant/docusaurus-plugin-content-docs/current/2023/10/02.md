## [輪胎粉塵佔海洋微塑膠的大部分](https://www.thedrive.com/news/tire-dust-makes-up-the-majority-of-ocean-microplastics-study-finds)

- 最近的一項研究表明，海洋中的大多數微塑膠來自輪胎粉塵，這會損害海洋生物和潛在的人類健康。
- 該研究還發現，由於電動汽車（EV）的重量和高扭矩，輪胎粉塵的排放甚至更高。
- 監管機構正在通過實施輪胎和制動器排放標準來解決這個問題，而製造商則研究替代輪胎成分和捕獲顆粒輸出的方法。

### [反應](https://news.ycombinator.com/item?id=37726539)

- 對話討論了與運輸環境影響相關的多個主題，強調輪胎粉塵是海洋中微塑膠的重要來源，以及基於汽車的運輸對空氣和表面污染的不利影響。
- 它探討了在郊區沒有私家車生活的可能性和優勢，質疑郊區基礎設施的可持續性，並研究了不同能源，特別是電動汽車（EV）對環境的影響。
- 討論的一個獨特方面包括可能使用蒲公英作為輪胎生產的橡膠的可持續來源，圍繞汽車與公共交通的辯論，基於重量的稅收以及對燃油效率低下的車輛的擔憂。

## [再見整數，你好UUIDv7](https://buildkite.com/blog/goodbye-integers-hello-uuids)

- Buildkite發佈了一個更新，引入了一個新產品，即登錄博客支援，以及關鍵改進，其中結合了用於數據存儲的UUIDv7作為其主要密鑰。
- UUIDv7 是一種唯一標識符系統，由於其來自順序鍵和 UUID 鍵的有益屬性而被使用。它對標識碼中的時間戳進行編碼，從而提高資料庫性能，無需協調標識碼生成並簡化應用程式邏輯。
- 該公司正在考慮通過分片其最大的Postgres資料庫來擴展其改進，並可能在未來使用UUIDv8，儘管由於其整體優勢，存儲略有增加。

### [反應](https://news.ycombinator.com/item?id=37733036)

- 討論集中在分散式系統中UUID（通用唯一標識符）的使用上，對其安全性和實用性有不同的意見，以及對Cuid2和UUIDv4等替代方案的建議。
- 隨機ID對系統性能、資料庫分片以及數據局部性與查詢效率的關係的影響也成為話題，表明瞭設計和優化分散式系統的複雜性。
- 最後，對話探討了不同版本的 UUID，如 UUIDv7、ULID 和 KSUID、它們的優點和局限性，以及在 GIS（地理資訊系統）應用程式中使用唯一 ID，強調了對更短 ID 表示的需求。

## [Influxdb 從 Go 切換到 Rust](https://old.reddit.com/r/rust/comments/16v13l5/influxdb_officially_made_the_switch_from_go_rust/)

- InfluxDB 是一個開源時間序列資料庫，由於其更高的性能、更好的錯誤處理、併發性以及與其他開源應用程式的改進集成，它幾乎將所有代碼 （99.5%） 從 Go 轉換為 Rust。
- 重寫花了大約 3 年的時間，旨在滿足無限基數、增強的分析查詢以及與 SQL 和更大生態系統的相容性等要求。
- 向 Rust 的過渡導致新版本 InfluxDB 在其雲環境中的性能顯著提高，討論還包括開發過程、DataFusion 查詢引擎的使用以及與 TimescaleDB 和 ClickHouse 的比較。

### [反應](https://news.ycombinator.com/item?id=37725778)

- 時間序列資料庫InfluxDB已經開始從Go程式設計語言過渡到Rust，引發了用戶的興趣和關於重寫軟體的好處的辯論。
- 由於InfluxDB的擴展問題，一些用戶已經轉向TimescaleDB和ClickHouse等替代資料庫，而另一些使用者則稱讚TimescaleDB的好處，如HA，備份和管理。
- 儘管開發了InfluxDB v3，但InfluxDB v1由於其簡單性和相容性而繼續廣泛使用。切換到 Rust 的決定是出於潛在的性能增強和 Rust 編譯器，分別滿足了開發人員和 InfluxData 團隊的懷疑和樂觀。

## [學習和測試 DMARC](https://www.learndmarc.com)

- “學習和測試 DMARC”控制台直觀地描述了電子郵件伺服器通信，並向使用者介紹了 SPF（寄件者策略框架）、DKIM（域名金鑰識別郵件）和 DMARC（基於域的郵件身份驗證、報告和一致性）。
- 用戶可以先向分配的位址發送電子郵件，並可以選擇欺騙電子郵件、載入隨機示例、測試 DMARC 知識和粘貼電子郵件標題。
- 該平台保證它僅將使用者的電子郵件用於DMARC設置檢查，並且不存儲任何用戶數據。

### [反應](https://news.ycombinator.com/item?id=37729964)

- 論壇討論強調了DMARC在避免電子郵件欺騙方面的優勢和劣勢，與會者建議BIMI可以提高安全性併為值得信賴的通信授予視覺徽標。
- SPF 和 DKIM 被強調為容易受到電子郵件欺騙的影響，DMARC 依賴於它們成功的身份驗證，從而導致了關於風險管理和電子郵件身份驗證行業標準改進的對話。
- 雖然對話包括其他主題，如Apple的“隱藏我的電子郵件”服務，電子郵件錯誤，過時的JavaScript代碼以及TLS和SSL等協定的相關性，但重點是增強電子郵件安全性，防止欺騙。
