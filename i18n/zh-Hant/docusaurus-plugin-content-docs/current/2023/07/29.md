## [Cap'n Proto 1.0](https://capnproto.org/news/2023-07-28-capnproto-1.0.html)

- Cap'n Proto，一種消息傳遞協議，已經發佈了其C++參考實現的1.0版本。
- 該協定已在生產中使用多年，表明其可靠性。
- 此版本包括用於提高性能的優化。
- 已進行更改以允許取消 RPC。
- 版本 2.0 的計劃包括對 C++ API 和 KJ C++ 工具包庫的修訂，以增強開發人員體驗。
- 這些更改將需要具有 C++20 支援的編譯器，並且可能使 KJ 對協程更加友好。
- 不會對序列化格式或 RPC 協定進行向後不相容的更改。
- 正式的2.0版本預計在幾年內不會發佈。

- 探索了Cap『n Proto進一步發展和改進的潛力。
- 對話涉及諸如軟體開發中的必填和可選欄位、靜態類型以及協定設計的挑戰等主題。
- 其他序列化格式（如Avro、protobuf和flatbuffers）與 Cap'n Proto 進行了比較。

## [Python 即將推出的無 GIL 模式](https://lwn.net/Articles/939568/)

- Python 指導委員會正在考慮一項提案，使全域解釋器鎖 （GIL） 在 CPython 中可選。
- 這將在Python中啟用無GIL模式，從而可能提高併發性。
- 該委員會正在謹慎行事，以確保向後相容性並避免Python 2和3拆分情況。
- 他們正在討論 ABI 相容性的要求，並解決對向後相容性的影響。
- 一些評論員對潛在的相容性問題表示擔憂，並推薦了增強Python併發性的替代方法。

### [反應](https://news.ycombinator.com/item?id=36914401)

- Python正在引入No-GIL（全域解釋器鎖定）模式，正如 lwn.net 上的一篇文章所證實的那樣。
- 全域解釋器鎖是 Python 中的一種機制，一次只允許一個線程執行 Python 位元組碼。
- No-GIL 模式旨在消除此限制，並允許多個線程同時執行 Python 位元組碼。
- 有關 No-GIL 模式的資訊最初是在 ycombinator.com 上共用的。
