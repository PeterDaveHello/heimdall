## [วิธีการสิ้นเปลืองแบนด์วิดท์ พลังงานแบตเตอรี่ และทำให้ผู้ดูแลระบบรำคาญ](https://rachelbythebay.com/w/2024/06/28/fxios/)

- พฤติกรรมนี้ทำให้เซิร์ฟเวอร์เกิดความเครียดและอาจเป็นที่น่ารำคาญอย่างยิ่งสำหรับผู้ดูแลระบบที่จัดการเซิร์ฟเวอร์เหล่านี้
- ตัวอย่างที่ให้มาแสดงให้เห็นถึงคำขอที่เหมือนกันหลายครั้งและคำขอ favicon ที่ไม่ถูกต้อง ซึ่งแสดงให้เห็นถึงปัญหาและผลกระทบต่อทรัพยากรของเซิร์ฟเวอร์

- ผู้ใช้เน้นปัญหาต่างๆ เช่น ประสิทธิภาพที่ช้า ข้อบกพร่อง และความสำคัญของปลั๊กอินบล็อกโฆษณาอย่าง uBlock Origin ซึ่งช่วยเพิ่มประสบการณ์การท่องเว็บบนอุปกรณ์เคลื่อนที่อย่างมาก

## [การแยกย่อยไบต์โค้ด: การเปิดเผยช่องโหว่ด้านความปลอดภัยของ Lua ในเกม Factorio](https://memorycorruption.net/posts/rce-lua-factorio/)

- ช่องโหว่ในการใช้งาน Lua ของ Factorio ทำให้เซิร์ฟเวอร์ที่เป็นอันตรายสามารถรันโค้ดใดๆ บนเครื่องลูกข่ายได้ ซึ่งได้รับการแก้ไขในเวอร์ชันต่ำกว่า 1.1.101
- Lua ของเกม Factorio ซึ่งมีความสำคัญต่อการทำงานของเกมและม็อดต่าง ๆ นั้น มีความเสี่ยงต่อเครือข่ายเนื่องจากโหมดผู้เล่นหลายคนแบบ deterministic lockstep
- เส้นทางการโจมตีนี้เกี่ยวข้องกับการโฮสต์เซิร์ฟเวอร์ที่มีโค้ด Lua ที่เป็นอันตราย ซึ่งนำไปสู่การรั่วไหลของหน่วยความจำและการรันโค้ดระยะไกลผ่านการจัดการไบต์โค้ดของ Lua

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40830005)

- การอภิปรายมุ่งเน้นไปที่ความเสี่ยงด้านความปลอดภัยของการรัน Lua bytecode ในเกม Factorio โดยเน้นถึงความจำเป็นในการมีเอกสารที่ดีกว่าและการใช้ sandboxing
- Factorio ได้ปิดการโหลด bytecode และจำกัดการใช้ไลบรารี debug เนื่องจากความกังวลด้านความปลอดภัย พร้อมกับมีข้อเสนอให้มีการสลับโหมด "Reduced Security Mode" สำหรับผู้ใช้ขั้นสูง
- ผู้เข้าร่วมแนะนำมาตรการรักษาความปลอดภัยต่างๆ เช่น การใช้การแยก VM การหลีกเลี่ยงไบต์โค้ดในระบบที่ไม่ฝังตัว และการตรวจสอบไบต์โค้ดให้แน่ใจ เช่นเดียวกับสภาพแวดล้อมของ Java และ .NET

## [คำสดุดีสำหรับ DevOps](https://matduggan.com/a-eulogy-for-devops/)

- DevOps ซึ่งเริ่มต้นด้วยความหวังว่าจะทำให้การพัฒนาและการดำเนินงานของซอฟต์แวร์เป็นไปอย่างราบรื่น มักนำไปสู่ความเสี่ยงที่รวมศูนย์และความล่าช้าเนื่องจากความท้าทายในการสื่อสารและการประสานงาน
- การเปลี่ยนไปสู่ DevOps มีเป้าหมายเพื่อลดการพึ่งพาบุคลากรทางเทคนิคเฉพาะทางและอำนวยความสะดวกในการเปลี่ยนแปลงในนาทีสุดท้าย แต่กลับส่งผลให้ระบบมีความซับซ้อนและมีค่าใช้จ่ายสูง แม้จะมีการนำคอนเทนเนอร์และโซลูชัน SaaS เช่น Datadog มาใช้ก็ตาม
- อุตสาหกรรมกำลังเปลี่ยนไปสู่การวิศวกรรมแพลตฟอร์มและกระบวนการทำงานที่ง่ายขึ้น โดยหันเหจากเทคโนโลยีที่ซับซ้อนอย่าง Kubernetes เพื่อมุ่งเน้นที่ความเรียบง่าย ความเสถียร และความคาดหวังในการเติบโตที่เป็นจริง

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40826236)

- โพสต์นี้กล่าวถึงการลดลงของ DevOps ที่รับรู้ได้ โดยเน้นว่าการมุ่งเน้นเป้าหมายเริ่มต้นที่ความเร็วและประสิทธิภาพมักนำไปสู่การปฏิบัติที่วุ่นวายและไม่ยั่งยืน
- มันเน้นการเปลี่ยนไปสู่การผสานรวมและการปรับใช้แบบต่อเนื่อง (CI/CD) และการพัฒนาบน trunk-based ซึ่งมีเป้าหมายเพื่อทำให้กระบวนการปรับใช้ง่ายขึ้นและลดความเสี่ยง
- การสนทนานี้สะท้อนถึงบทบาทที่เปลี่ยนแปลงไปในทีมเทคโนโลยี เช่น วิศวกรความน่าเชื่อถือของไซต์ (SREs) และวิศวกรแพลตฟอร์ม ซึ่งบ่งบอกถึงการเคลื่อนย้ายออกจากบทบาท DevOps แบบดั้งเดิม

## [การเข้ารหัส AEAD แบบ XAES-256-GCM ที่มี nonce ขยาย](https://words.filippo.io/dispatches/xaes-256-gcm/)

- XAES-256-GCM เป็นอัลกอริธึม AEAD (Authenticated Encryption with Associated Data) ใหม่ที่ใช้กุญแจขนาด 256 บิตและ nonce ขนาด 192 บิต ออกแบบมาเพื่อความปลอดภัย การปฏิบัติตามมาตรฐาน FIPS 140 และการนำไปใช้ที่ง่ายดาย
- มันเป็นโครงสร้าง nonce ที่ขยายบน AES-256-GCM ซึ่งต้องการการเรียกใช้ AES-256 สามครั้งต่อข้อความ โดยหนึ่งครั้งสามารถคำนวณล่วงหน้าได้ และได้รับการสนับสนุนโดยไลบรารีการเข้ารหัสทั่วไปและ API ของ OpenSSL
- มีการใช้งานจากบุคคลที่สามสำหรับ .NET 8+, pyca/cryptography และ Web Cryptography API โดยการใช้งานอ้างอิงของ Go มีความยาวไม่เกิน 100 บรรทัดโดยใช้ไลบรารีมาตรฐาน

- OpenLoco ยังคงใช้ไบนารีดั้งเดิมอย่างมาก ซึ่งทำให้เกิดคำถามเกี่ยวกับการใช้ไลบรารี 64 บิตและกลไกสมัยใหม่
