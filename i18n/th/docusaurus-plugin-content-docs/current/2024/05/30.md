# 2024-05-30

## [ทําไมฉันถึงย้ายออกจาก GraphQL หลังจากหกปี](https://bessey.dev/blog/2024/05/24/why-im-over-graphql/)

- ผู้เขียนแบ่งปันประสบการณ์หกปีกับ GraphQL โดยเริ่มแรกสังเกตเห็นประโยชน์ของมันเหนือ JSON REST API ที่ไม่ได้พิมพ์ แต่ในที่สุดก็ชี้ให้เห็นข้อเสียที่สําคัญ
- ปัญหาสําคัญของ GraphQL ได้แก่ พื้นผิวการโจมตีที่เพิ่มขึ้นการอนุญาตที่ซับซ้อนความท้าทายในการ จํากัด อัตราช่องโหว่การแยกวิเคราะห์แบบสอบถามและปัญหาด้านประสิทธิภาพเช่นปัญหา N + 1
- ผู้เขียนแนะนําให้พิจารณาทางเลือกอื่น เช่น JSON REST API ที่สอดคล้องกับ OpenAPI และเครื่องมือที่ทันสมัย เช่น FastAPI, tsoa และ TypeSpec เพื่อการพัฒนา API ที่มีประสิทธิภาพและปลอดภัยยิ่งขึ้น

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40521518)

- หลังจากหกปีของการใช้ GraphQL ผู้เขียนพบว่ามันซับซ้อนเกินไป โดยเฉพาะอย่างยิ่งในด้านการอนุญาต ประสิทธิภาพ และการดีบัก ซึ่งแนะนําว่าปลายทาง REST แบบดั้งเดิมอาจมีประสิทธิภาพมากกว่า
- การอภิปรายเน้นย้ําถึงความท้าทายของการรวม GraphQL เข้ากับระบบแบ็กเอนด์ โดยเน้นย้ําถึงความจําเป็นในการมีความรู้และเอกสารโดยละเอียด และเสนอโซลูชัน เช่น ตัวสร้างคิวรีฝั่งเซิร์ฟเวอร์และการออกแบบสคีมาเป็นอันดับแรก
- การอภิปรายเปรียบเทียบความยืดหยุ่นและการสืบค้นแบบซ้อนที่มีประสิทธิภาพของ GraphQL กับความเรียบง่ายและความน่าเชื่อถือของ REST โดยสังเกตว่าตัวเลือกระหว่างพวกเขาควรขึ้นอยู่กับความต้องการและวุฒิภาวะเฉพาะของโครงการ

## [FrankenPHP: เซิร์ฟเวอร์ PHP ที่ทันสมัยพร้อมคุณสมบัติ Go, Caddy และ Cloud-Native](https://frankenphp.dev/)

- FrankenPHP เป็นเซิร์ฟเวอร์แอปพลิเคชัน PHP ที่ทันสมัยที่เขียนใน Go ซึ่งรวมตัวดําเนินการ PHP อย่างเป็นทางการเข้ากับเว็บเซิร์ฟเวอร์แคดดี้ และรองรับ HTTP/1.1, HTTP/2, HTTP/3 และใบรับรอง HTTPS อัตโนมัติ
- คุณสมบัติที่สําคัญ ได้แก่ โหมดผู้ปฏิบัติงานเพื่อประสิทธิภาพที่เพิ่มขึ้นความเข้ากันได้กับเฟรมเวิร์ก PHP ยอดนิยมรองรับรูปแบบการบีบอัดที่ทันสมัยเมตริก Prometheus ในตัวการบันทึกที่มีโครงสร้างและการจัดการเหตุการณ์แบบเรียลไทม์ผ่านฮับ Mercure
- ออกแบบมาสําหรับสภาพแวดล้อมแบบ Cloud-Native FrankenPHP นั้นง่ายต่อการปรับใช้เป็นอิมเมจ Docker หรือไบนารีแบบสแตนด์อโลนนําโดย Kévin Dunglas ด้วยการออกแบบโดย Laury Sorriaux และการสนับสนุนจาก Les-Tilleuls.coop

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40519722)

- FrankenPHP เป็นเซิร์ฟเวอร์แอปพลิเคชัน PHP ที่ทันสมัยซึ่งรวม PHP ไว้ในเซิร์ฟเวอร์ Go ทําให้การปรับใช้ง่ายขึ้นในไบนารีเดียวดึงดูดนักพัฒนาด้วยการออกแบบที่ใช้งานง่าย

- แม้จะมีศักยภาพ แต่ FrankenPHP ก็เผชิญกับการวิพากษ์วิจารณ์เกี่ยวกับกระบวนการสร้างที่ซับซ้อนปัญหาด้านประสิทธิภาพและการขาดการสนับสนุนโหมดผู้ปฏิบัติงานซึ่งบ่งบอกถึงความจําเป็นในการปรับแต่งเพิ่มเติมและเอกสารที่ดีขึ้น

## [จดหมายรักที่ยังไม่ได้เปิดของ Richard Feynman ถึงภรรยาผู้ล่วงลับเปิดเผยหลังจากการเสียชีวิตของเขา](https://lettersofnote.com/2012/02/15/i-love-my-wife-my-wife-is-dead/)

- Richard Feynman นักฟิสิกส์ชื่อดัง เขียนจดหมายรักจากใจจริงถึง Arline ภรรยาผู้ล่วงลับของเขา 16 เดือนหลังจากที่เธอเสียชีวิตในปี 1945
- จดหมายซึ่งยังไม่ได้เปิดจนกระทั่งหลังจากการเสียชีวิตของ Feynman ในปี 1988 แสดงถึงความรักและความเศร้าโศกที่ยั่งยืนของเขา
- จดหมายที่สะเทือนใจนี้มีอยู่ในหนังสือ "จดหมายบันทึก"

- การอภิปรายเจาะลึกถึงความซับซ้อนทางอารมณ์และศีลธรรมของการแสวงหาความสัมพันธ์ใหม่หลังจากการเสียชีวิตของคู่สมรสโดยเน้นความยินยอมที่ได้รับแจ้งและไม่สามารถถูกแทนที่ได้ของความสัมพันธ์ระยะยาวที่ลึกซึ้ง
- โพสต์ Hacker News ทบทวนจดหมายของ Feynman ถึงภรรยาผู้ล่วงลับของเขาจุดประกายการถกเถียงเกี่ยวกับความรักการสูญเสียจริยธรรมในการเผยแพร่งานเขียนส่วนตัวและหัวข้อที่กว้างขึ้นเกี่ยวกับบทบาทของความตายในวิวัฒนาการทางสังคมและการเติบโตส่วนบุคคล

## [หลีกเลี่ยงสิ่งที่เป็นนามธรรมก่อนเวลาอันควร: ทบทวนหลักการ DRY ในการเข้ารหัส](https://testing.googleblog.com/2024/05/dont-dry-your-code-prematurely.html)

- Dan Maksimovich แนะนําว่าอย่าใช้หลักการ "Don't Repeat Yourself" (DRY) ก่อนวัยอันควรในการเขียนโค้ด เนื่องจากอาจนําไปสู่สิ่งที่เป็นนามธรรมก่อนเวลาอันควรและทําให้การเปลี่ยนแปลงในอนาคตซับซ้อนขึ้น
- เขาแนะนําให้อดทนต่อการทําซ้ําโค้ดบางอย่างในขั้นต้นเพื่อรักษาตรรกะเฉพาะบริบทที่ชัดเจนยิ่งขึ้น และอํานวยความสะดวกในการแก้ไขในอนาคตที่ง่ายขึ้น
- แนวทางของ Maksimovich สอดคล้องกับหลักการ "You Aren't Gonna Need It" (YAGNI) ซึ่งแนะนําให้หลีกเลี่ยงการเพิ่มฟังก์ชันที่ไม่จําเป็นจนกว่าจะจําเป็นจริงๆ

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40525064)

- การอภิปรายเน้นย้ําถึงความจําเป็นในการสร้างสมดุลระหว่างความสามารถในการอ่านโค้ดและการทํางานร่วมกันกับหลักการ "Don't Repeat Yourself" (DRY) ในการพัฒนาซอฟต์แวร์
- ผู้ร่วมให้ข้อมูลยอมรับว่าแม้ว่า DRY จะช่วยเพิ่มการทํางานร่วมกัน แต่ก็อาจทําให้การดีบักซับซ้อนขึ้นเนื่องจากสิ่งที่เป็นนามธรรมที่ซ้อนกัน
- การสนทนาเน้นย้ําถึงความสําคัญของการตัดสินใจตามบริบทและการแลกเปลี่ยนในด้านวิศวกรรมโดยเน้นตัวอย่างที่ใช้ได้จริงและข้อมูลเชิงลึกทางประวัติศาสตร์เกี่ยวกับการจัดการหนี้ทางเทคนิคและเพิ่มประสิทธิภาพด้านประสิทธิภาพที่สําคัญ

## [กลไกความสนใจใหม่เหนือกว่าความสนใจหลายหัวมาตรฐานในด้านประสิทธิภาพและประสิทธิภาพ](https://arxiv.org/abs/2403.01643)

- บทความ "You Need to Pay Better Attention" โดย Mehran Hosseini และ Peyman Hosseini แนะนํากลไกความสนใจใหม่สามประการ: Optimised Attention, Efficient Attention และ Super Attention
- Optimised Attention จะลดพารามิเตอร์ลง 25% และต้องการการคูณเมทริกซ์น้อยลงหนึ่งครั้งต่อหัว ในขณะที่ Efficient Attention จะลดพารามิเตอร์ลงครึ่งหนึ่งและลดการคูณเมทริกซ์ลงสองรายการต่อหัว ซึ่งเพิ่มความเร็วเป็นสองเท่า
- Super Attention มีประสิทธิภาพเหนือกว่าความสนใจมาตรฐานในงานประมวลผลการมองเห็นและภาษาธรรมชาติอย่างมีนัยสําคัญ โดยประเมินจากชุดข้อมูล เช่น MNIST, CIFAR100, IMDB Movie Reviews และ Amazon Reviews

### [ปฏิกิริยา](https://news.ycombinator.com/item?id=40515957)

- การวิจัยล่าสุดแนะนํากลไกความสนใจใหม่ๆ เช่น Optimised Attention, Efficient Attention, Super Attention และ Infini-attention ซึ่งมีประสิทธิภาพเหนือกว่าความสนใจแบบหลายหัวมาตรฐานในด้านประสิทธิภาพและประสิทธิภาพ
- 2D Discrete Fourier Transform (DFT) กําลังถูกสํารวจเพื่อทดแทนกลไกความสนใจ โดยมีการอภิปรายเกี่ยวกับการรวมเข้ากับโครงข่ายประสาทเทียมและการเพิ่มประสิทธิภาพที่ระบุไว้ในกระดาษ FNet
- การสนทนาเน้นย้ําถึงความซับซ้อนของระบบ AI ผลกระทบต่อสิ่งแวดล้อมของโมเดลขนาดใหญ่ และความเกลียดชังความเสี่ยงของอุตสาหกรรม โดยเน้นย้ําถึงความจําเป็นในการทดลองเพิ่มเติมและการตรวจสอบความถูกต้องของเทคนิคใหม่ๆ
