# 2023-12-19

## [Figma 和 Adobe 在监管审查后放弃合并](https://www.figma.com/blog/figma-adobe-abandon-proposed-merger/)

- 经过漫长的监管审查程序后，Figma 和 Adobe 取消了合并计划。
- Figma 将保持其独立地位，并愿意探索与 Adobe 合作的机会。
- Figma 在过去的 15 个月里取得了长足的进步，重点是简化数字产品的设计和开发。首席执行官迪伦-菲尔德对公司的未来表示乐观，并对社区的支持表示感谢。

### [反应](https://news.ycombinator.com/item?id=38681861)

- Figma 和 Adobe 之间的潜在合并已被取消，这引发了关于合并原因及其对 Figma 员工影响的讨论。
- 对话中探讨了 Figma 等设计工具的优势和局限性，以及人工智能对设计行业的潜在颠覆。
- 讨论还涉及初创企业文化中的合并和首次公开募股、监管监督、长期可持续性以及反垄断法在数字市场中的作用。

## [Wasm3 在俄罗斯入侵中进入维护阶段](https://github.com/wasm3/wasm3)

- WebAssembly 解释器 Wasm3 的新功能开发暂时停止，因为开发人员的房子被入侵其国家的俄罗斯人摧毁了。
- 开发人员将继续维护项目，审查并合并 Pull Request。
- Wasm3 是一种多功能解释器，可用于各种平台和架构，并通过了 WebAssembly 规范测试套件。它在可移植性、安全性、可执行文件大小、内存使用和启动延迟等方面具有优势。它还具有在嵌入式设备上运行 WebAssembly 的潜在应用。该项目采用 MIT 许可发布。

### [反应](https://news.ycombinator.com/item?id=38681672)

- 对话深入探讨了与俄乌冲突有关的各种话题，如乌克兰历史、民族主义、宣传、资源开采、经济制裁和核武器。
- 其中包括关于是否支持乌克兰的辩论、审视俄罗斯的意图以及讨论不同编程语言的有效性。
- 对话还讨论了减少代码中资源泄漏的技术，并展示了有关当前冲突及其后果的各种观点和关切。

## [作者的 MacBook 被锁，蒂姆-库克为其解锁](https://www.tokyodev.com/articles/unbricking-my-macbook-took-an-email-to-tim-cook)

- 提交人的 MacBook 在返还给他们时被加上了激活锁，无法访问该设备。
- 苹果公司最初拒绝解锁 MacBook，但作者通过电子邮件联系了蒂姆-库克，最终解锁了 MacBook。
- 该 MacBook 已被新创建的 iCloud 账户抹除并报告丢失，这表明它可能被带到商店重置并报告丢失以勒索钱财。
- 作者强调，苹果公司有必要修改其设置工作流程，以更好地传达不设置 FindMy 的后果，因为不法商店可能会利用这种情况。
- 作者向苹果安全研究部门报告了这个问题，但被驳回。

### [反应](https://news.ycombinator.com/item?id=38691025)

- 讨论围绕设备安全、所有权和客户支持展开，重点是苹果产品。
- 用户对苹果公司控制他们的设备表示不满，但也有一些用户对防盗功能表示赞赏。
- 建议加强安全措施，并在防盗和用户需求之间找到平衡。

## [大众汽车回应投诉，在汽车中重新引入物理按钮](https://www.thedrive.com/news/vw-is-putting-buttons-back-in-cars-because-people-complained-enough)

- 大众汽车正通过在汽车内饰中重新引入实体按键来回应客户和评论家对令人沮丧的触摸屏控制的抱怨。
- 这一决定是根据反馈意见做出的，目的是解决当前触摸滑块的不稳定性和非功能性问题。
- 暖通空调和窗户功能增加了物理控制，有望提升整体用户体验，同时触摸屏仍将存在。

### [反应](https://news.ycombinator.com/item?id=38686967)

- 争论的焦点是在汽车和电器中选择物理按钮还是触摸屏，用户对每种选择都表达了自己的偏好。
- 用户喜欢物理按键带来的直观、一致的体验，而其他人则看重触摸屏的简洁和时尚。
- 设计不佳的界面让用户感到沮丧，他们强调需要在美观和功能之间取得平衡。他们还提到了马自达和特斯拉等特定车型，用户呼吁采用可定制的用户友好界面。最终，与会者一致认为，消费者的反馈和偏好应推动汽车行业的设计决策。

## [与 GCC 集成的 Rust 编译器进展情况](https://lwn.net/SubscriberLink/954787/41470c731eda02a4/)

- gccrs 项目正在努力创建一个 Rust 编译器，以纳入 GCC 14 版本，目标版本为 Rust 1.49。
- 该项目目前的重点是支持核心和分配板条箱的编译，但在宏解析和装饰宏方面面临挑战。
- 该项目旨在使 Rust 代码能够使用 GCC 进行编译，从而促进 Rust 的发展并支持嵌入式系统。另一个高级项目 rustc_codegen_gcc，已经可以在不打补丁的情况下为 Linux 编译 Rust，讨论内容包括将 Polonius 集成到 Rust 编译器中，以及实施基于 GCC 的 Rust 编译器的好处，如错误暴露和引导。

### [反应](https://news.ycombinator.com/item?id=38684102)

- 科技界正在讨论开发基于 GCC 的 Rust 编译器 gccrs。
- 关于多个 Rust 实现的争论不绝于耳，一些人认为这有助于发现错误和改进语言，而另一些人则担心兼容性和重复性。
- 编译器在跟上语言标准方面面临挑战，而使用多种编译器有助于提高代码质量和检测错误。对话还探讨了 Rust 正式规范的必要性以及标准化的潜在好处和缺点。
