## [π в других вселенных](https://azeemba.com/posts/pi-in-other-universes.html)

- В статье рассматривается определение значения пи, фундаментальной математической константы, и то, как на него влияет определение расстояния.
- В нем вводятся такие понятия, как евклидово и манхэттенское расстояния, метрики, влияющие на вычисление пи.
- В нем отмечается, что наименьшее возможное значение pi по всем метрикам равно 3,14159, однако по многим другим метрикам pi колеблется в пределах от 3 до 4. Это говорит о том, что значение pi не является абсолютным, а зависит от таких факторов, как выбранная метрика.

### [Реакции](https://news.ycombinator.com/item?id=38063536)

- Обсуждаются различные аспекты математики, в том числе ее интерпретация как логической игры, применимость и масштабируемость.
- Беседа проливает свет на происхождение математики и ее взаимосвязь с наукой, а также на навигацию по новым и эзотерическим математическим понятиям.
- В диалоге также размышляется о дискреционном характере математических аксиом, о следствиях из числа "пи", об идее многообразия констант окружности, а также о необходимости прочтения обширного трактата по теории мер.

Автор предлагает начинать с монолитного подхода и переходить к микросервисам по мере необходимости, при наличии достаточного опыта и отлаженной платформы.

### [Реакции](https://news.ycombinator.com/item?id=38069915)

- В центре внимания - компромиссы при использовании микросервисов по сравнению с монолитной архитектурой, в том числе технические инструменты, такие как непрерывная интеграция/непрерывное развертывание (CI/CD), потенциальные сложности с базами данных и необходимость грамотного планирования.
- Обсуждается, что, хотя микросервисы дают такие преимущества, как повышение контроля над отдельными командами и ускорение разработки программного обеспечения, они также могут создавать проблемы, требующие глубокого понимания зависимостей существующей системы.
- В ходе обсуждения подчеркивается, что решение о внедрении архитектуры микросервисов должно приниматься после тщательной оценки и под сильным руководством, чтобы эффективно управлять сложностями.

## [Kubernetes выпускает инструмент ingress2gateway для предстоящего GA API шлюза](https://kubernetes.io/blog/2023/10/25/introducing-ingress2gateway/)

- В блоге Kubernetes представлен "ingress2gateway" - инструмент, помогающий перейти от Ingress к более функциональному и гибкому Gateway API для управления доступом к внешним сервисам в кластерах Kubernetes.
- В блоге были разобраны преимущества Gateway API перед Ingress - ролевая стратегия, переносимость, выразительность и расширяемость.
- В нем также приводится руководство по установке и использованию ingress2gateway для миграции. В завершение блога было отмечено постоянное развитие Gateway API и сообщество, поддерживающее его.

### [Реакции](https://news.ycombinator.com/item?id=38062269)

- В рамках подготовки к выпуску General Availability (GA) Gateway API компания Kubernetes представила инструмент ingress2gateway, призванный упростить использование API для специалистов DevOps и SRE.
- Инструмент, разработанный компанией Kong - известным поставщиком Ingress, поддерживает как Ingress, так и Gateway API, обеспечивая плавный переход между ними.
- Этот шаг связан с ограничениями, возникшими в Ingress API при работе с такими расширенными функциями, как маршрутизация строк запросов и взвешенная балансировка нагрузки, которые, как предполагается, будут реализованы в Gateway API.

## [Перенос нашего бэкенда с Vercel на Fly.io](https://www.openstatus.dev/blog/migration-backend-from-vercel-to-fly)

- В статье рассказывается о переносе бэкенда OpenStatus с Vercel на Fly.io, что было вызвано необходимостью создания более легкого сервера и опасениями по поводу цен на Vercel.
- В ходе перехода пришлось столкнуться с такими проблемами, как оптимизация Docker-образа в монорепо, решение проблемы таймаутов развертывания на Fly.io и устранение ошибки в среде выполнения Bun.
- Несмотря на эти препятствия, переход на Fly.io прошел успешно, и авторы по-прежнему высоко оценивают продукт компании Vercel.

### [Реакции](https://news.ycombinator.com/item?id=38062649)

- Обсуждается опыт и мнения пользователей о переносе бэкенда с Vercel на Fly.io, подчеркиваются проблемы с ненадежностью развертывания, функциональностью и ценообразованием Vercel.
- Влияние Vercel на React, экономическая эффективность edge и лямбда-функций, а также стабильность платформы Fly.io - важные подтемы в рамках дискуссии.
- В ходе дискуссии также упоминаются альтернативные платформы, такие как DigitalOcean, Render и Hetzner, и обсуждается компромисс между краткосрочной выгодой и долгосрочными эксплуатационными расходами при развертывании инфраструктуры.
